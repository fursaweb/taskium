# Предложенные задачи по итогам обзора кодовой базы

## 1) Исправление опечатки
**Задача:** Исправить текст ошибки валидации `Title must be at least 3 charachters long` на корректное `Title must be at least 3 characters long`.

**Почему это важно:** сообщение отображается пользователю/разработчику и содержит орфографическую ошибку, что снижает качество интерфейса и доверие к продукту.

**Где найдено:** `frontend/app/api/tasks/route.ts`.

---

## 2) Исправление ошибки в логике/безопасности
**Задача:** Ограничить обновление и удаление задач проверкой владельца (по `userId`), чтобы нельзя было изменить/удалить задачу другого пользователя по одному только `id`.

**Почему это важно:** сейчас в `PUT` и `DELETE` используются запросы по `id` без фильтра по пользователю, что открывает возможность несанкционированного изменения данных.

**Где найдено:** `frontend/app/api/tasks/route.ts` (обновление), `frontend/app/api/tasks/[id]/route.ts` (удаление).

---

## 3) Исправление несоответствия документации
**Задача:** Обновить `frontend/README.md`: добавить шаг запуска backend-сервера и уточнить, что фронтенд зависит от API по адресу `http://localhost:5000/api/v1`.

**Почему это важно:** текущая инструкция в README описывает только запуск Next.js, но HTTP-клиент фронтенда жёстко настроен на отдельный backend. Без него приложение частично неработоспособно.

**Где найдено:** `frontend/README.md`, `frontend/http/index.ts`.

---

## 4) Улучшение тестового покрытия
**Задача:** Добавить интеграционный тест на авторизацию и владение ресурсом для API задач: проверка, что пользователь не может обновить/удалить чужую задачу и получает `401/403`.

**Почему это важно:** тест зафиксирует критичное поведение безопасности и предотвратит регрессии при дальнейших изменениях API.

**Где применить:** тесты для обработчиков в `frontend/app/api/tasks/route.ts` и `frontend/app/api/tasks/[id]/route.ts`.

---

## Дополнительные ошибки в логике/безопасности (к исправлению)

### 5) Сервер доверяет `userId` из клиентского запроса
**Задача:** На backend (Express) убрать доверие к `userId` из `req.query`/`req.body`; получать идентификатор пользователя только из проверенного механизма аутентификации (например, middleware с верификацией токена).

**Риск:** сейчас любой клиент может подменить `userId` и читать/изменять чужие задачи.

**Где найдено:** `server/controllers/task-controller.js` (`getAllTasks` использует `req.query.userId`), `server/models/tasks.js` (поле `userId` принимается как обычная строка).

### 6) Отсутствует авторизация на backend-маршрутах задач
**Задача:** Добавить middleware авторизации на маршруты `/api/v1/tasks` и проверку прав доступа в `getTask`, `updateTask`, `deleteTask`.

**Риск:** API доступен без проверки личности, что допускает несанкционированный доступ к данным.

**Где найдено:** `server/routes/index.js`, `server/controllers/task-controller.js`.

### 7) Mass assignment при обновлении задачи
**Задача:** В `updateTask` на backend разрешать обновление только whitelisted-полей (`title`, `description`, `date`, `isCompleted`, `isImportant`) и игнорировать/отклонять остальные.

**Риск:** передача `req.body` напрямую в `findByIdAndUpdate` позволяет клиенту менять поля, которые не должны изменяться (например, `userId`).

**Где найдено:** `server/controllers/task-controller.js`.
